# Правила проекта

Это шаблон, на основе которого запускаются новые проекты. Правила ниже действуют
по умолчанию для любой задачи в этом репозитории, если пользователь явно не попросил иначе.

## Дизайн и UI

- Дизайн должен выглядеть **современно и не как дефолтный шаблон** — избегай ощущения
  "стандартный shadcn-каркас как есть, без единой правки". У проекта должен быть свой
  визуальный акцент: осмысленная цветовая палитра (не голый серый/чёрно-белый minimal
  по умолчанию), продуманная типографика, уместные микро-анимации и переходы (hover,
  transition, motion), аккуратные состояния (loading/empty/error), а не только "happy path".
- Ориентируйся на актуальные тренды в UI (2025–2026): мягкие тени и glow вместо резких
  границ, плавные скругления, продуманная плотность интерфейса, тёмная тема не как
  автогенерация, а как отдельно выверенный вариант.
- Не переусердствуй: "необычно" не значит "нечитаемо" или "недоступно" — контраст,
  читаемость и доступность (a11y) важнее эффектности.
- Если для задачи предоставлена ссылка на Figma-макет — сверяйся с ним перед вёрсткой
  (через Figma MCP, если он подключён в сессии) вместо того, чтобы придумывать дизайн с нуля.

## Компоненты — shadcn/ui

- Не пиши UI-примитивы (Button, Input, Dialog, Select и т.п.) руками с нуля — сначала
  проверь `components/ui`, а если нужного компонента там нет, добавь его официально:
  ```bash
  npx shadcn@latest add <component>
  ```
  Это гарантирует единый стиль, доступность и совместимость с темой проекта.
- Кастомизируй уже добавленный компонент точечно (через `className`, `cva`-варианты),
  а не форкай его логику без необходимости.
